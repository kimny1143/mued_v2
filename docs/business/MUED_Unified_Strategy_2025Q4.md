## エグゼクティブサマリー

MUEDは**教育管理システム（LMS）から創造プロセス記録基盤（CMS）への進化過程**にあります。

本文書は、2025年9月から10月にかけて行われた戦略的議論と技術実装の整合性を統合し、今後12ヶ月の明確なロードマップを提示するものです。

### 核心的転換点（2025年10月29日）

- **ポジショニング変更:** 「教育SaaS」→「創造支援CMS」
- **収益モデル:** 単層B2Cサブスク → **二層サブスクモデル（B2C教育基盤 + B2B創造ログライセンス）**
- **技術哲学:** プラグイン過剰設計からの「焦点化」— 理念の後退ではなく**解像度の進化**

### 現在位置（2025年11月26日更新）

| 項目 | 実装状態 | 備考 |
|------|---------|------|
| LMS基本機能 | ✅ 実装済み | レッスン予約、Stripe課金、ユーザー管理 |
| note.com RSS連携 | ✅ Phase 1完了 | マガジンRSS取得、検索デバウンス実装済み |
| **MUEDnote Desktop** | ✅ **v3実装中** | **Tauri/Rust/local-first デスクトップアプリ** |
| プラグインアーキテクチャ | 🟡 簡易版のみ | 動的ローダーは見送り、直接インポート方式採用 |
| 双方向学習循環 | 🔴 未実装 | Phase 2実装対象 |
| 創造データ層 | 🔴 未実装 | Phase 3以降 |

### 製品構成（2025年11月26日確定）

```
MUED エコシステム
├── MUED LMS (Web)     ← Next.js 15.5 / Clerk / Vercel
│   教育プラットフォーム本体
│
└── MUEDnote (Desktop) ← Tauri v2 / Rust / local-first
    音楽学習・制作ノートアプリ
    詳細: apps/muednote-v3/README.md
```

**決定事項:**
- MUEDnote Web版POCは廃止 → `docs/archive/muednote-web-poc/`
- Desktop版に一本化（認証不要のlocal-first設計）
- MUED LMSとの連携はオプション機能として将来検討

## 2. 現行アーキテクチャ：二層構造

### Layer 1: LMS層（教育SaaS基盤）

**目的:** 事業としての説明可能性と収益安定化

**技術スタック:**
- Frontend: Next.js 15.5 (App Router), React 19, TailwindCSS 4
- Backend: Clerk認証, Neon PostgreSQL, Drizzle ORM
- Payments: Stripe
- AI: OpenAI API（教材生成）

**主要機能:**
- レッスン予約・管理
- 教材アクセス（note.com RSS連携含む）
- 学習進捗管理
- 簡易フィードバック

**KPI:**
- B2Cユーザー: 300〜500人
- 月次収益: 30万〜150万円（サブスクのみ）
- B2B契約: 初年度5件、次年度20件

### Layer 2: 創造データ層（理念拡張）

**目的:** 人間×AI共創過程の記録と再利用

**技術提案:**
```sql
-- 創造トレーステーブル
CREATE TABLE creation_trace (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  prompt TEXT NOT NULL,
  ai_response TEXT NOT NULL,
  human_modification TEXT,
  error_tags TEXT[],
  created_at TIMESTAMP DEFAULT NOW()
);

-- AI対話ログ
CREATE TABLE ai_dialogue_log (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  session_id UUID NOT NULL,
  message_history JSONB NOT NULL,
  context_tags TEXT[],
  created_at TIMESTAMP DEFAULT NOW()
);
```

**UX提案:**
- 反芻ダッシュボード（思考変遷の可視化）
- 誤差マップ（AI提案と人間出力の差分分析）
- RAG観測パネル（根拠提示率・応答正確度・コスト計測）

**実装状態:** 未実装（Phase 3以降）

## 5. 技術的決定事項（アーキテクチャ原則）

### 5.1 実装哲学：焦点化（Refocusing）

MUEDの進化は「理念の後退」ではなく「解像度の進化」です。

| 観点 | 初期構想 | 現行MVP | Phase 2以降 |
|------|----------|---------|-------------|
| 目的 | 創造過程の理解と記録 | 教育補助とレッスン管理 | 創造ログ×教育の統合 |
| AIの役割 | 創造の共作者 | 教材生成・学習補助 | 反芻対話・誤差分析 |
| データ | 思考と反芻のログ | 学習履歴と教材利用 | 創造トレース×学習履歴 |
| UX | 創造の軌跡を提示 | レッスン体験の最適化 | 反芻ダッシュボード |

### 5.2 プラグインアーキテクチャの簡略化決定

**問題点:**
- 動的`import()`がNext.js/Turbopackと非互換
- 過剰な依存性注入システムが保守コストを増大

**解決策:**
- `NoteContentFetcher`を直接インポート
- 必要に応じてファクトリーパターンで拡張
- プラグインローダーは将来的にStatic Import方式で再設計

**コード例（現行実装）:**
```typescript
// app/api/content/route.ts
import { NoteContentFetcher } from '@/lib/plugins/note/note-content-fetcher';

async function initializePlugins() {
  const noteFetcher = new NoteContentFetcher();
  registry.register('note', noteFetcher);
}
```

**再拡張トリガー条件:**
- 外部ソース連携が3件を超える
- ビルド時間が90秒を超える
- import循環依存が2件以上発生

上記いずれかに該当した場合、Static Import Plugin v2の設計を開始。

### 5.3 データ管理方針

**プロヴナンス（provenance）管理:**
- 全データに出所・権利・取得方法・許諾情報をメタデータで付与
- デジタル署名（C2PA等）の段階的導入

**RAG観測指標:**
- 根拠提示率（citation rate）
- 応答正確度（accuracy）
- 応答遅延（latency）
- トークン原価（cost per token）

**アクセス制御:**
- エージェント型アクセス（データソース側の認可を継承）
- Vector DBへの一括同期は避ける（セキュリティリスク）

#### 5.3.1 データ分類別メタデータ要件

| データ種別 | 必須メタデータ | 保管年限 | アクセス権限 | 外部提供 |
|-----------|--------------|---------|------------|---------|
| 教材 | source_uri, license_type, acquisition_method, rights_holder, permission_flag | 5年 | 契約ユーザーのみ | 許諾済みのみ |
| 創造ログ | user_id, session_id, prompt, ai_response, human_modification, consent_flag | 3年 | 本人 + 研究用（匿名化） | オプトイン必須 |
| 生成物 | origin_map（教材/ログ寄与率）, generation_params, hash_c2pa | 1年 | 本人のみ | 本人承諾時のみ |

## 6. リスクと対策

### 6.1 技術リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| AI API費用の高騰 | 収益圧迫 | トークン使用量の監視、キャッシュ戦略 |
| RAG精度不足 | UX低下 | ハイブリッド検索（Vector + BM25） |
| スケーラビリティ | 応答遅延 | Redis活用、DB最適化 |

### 6.2 事業リスク

| リスク | 影響 | 対策 |
|--------|------|------|
| B2B契約の遅延 | 収益未達 | B2Cサブスク基盤の強化 |
| サポート負荷 | 運用コスト増 | クローズドβ、FAQ自動化 |
| 法規制強化（EU AI Act等） | コンプライアンスコスト | データ透明性の先行実装 |

### 6.3 法務リスク

**著作権侵害リスク:**
- 日本著作権法30条の4対応（権利者意向尊重、市場侵害回避）
- 自社制作または許諾済みコンテンツに限定

**AI生成物の責任:**
- EU AI Actの動向追跡
- 生成物の出所・用途・責任を明確化

## 7. SWOT分析（2025年10月版）

### 強み（Strengths）
- 代表木村の音楽制作実績（紅白、ミリオンヒット）
- AI駆動開発による低コスト運用（月5〜10万円 vs 人力5,800万円）

### 弱み（Weaknesses）
- マーケティング・集客ノウハウ不足
- 個人開発体制（代表+AI）
- B2B営業経験の不足

### 機会（Opportunities）
- 音楽教育市場のデジタル化需要
- AI教材生成の市場優位性
- 補助金・研究機関連携の可能性

### 脅威（Threats）
- 大手EdTech企業の参入
- AI規制強化
- サブスク疲れ（市場飽和）
